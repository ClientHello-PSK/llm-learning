# 知识点 01：TP 基本原理

> 核心问题：单层算力/显存放不下时，怎么把一层切开跑在多张卡上？

---

## 1. TP 的物理类比

把一层矩阵乘法想象成"很多个独立计算单元"：

```
Y = X × W      X: [M, K]   W: [K, N]   Y: [M, N]
```

单卡算这个：算 N 个输出列，每列都是 X × W[:, j]。

TP 的核心想法：**让多张卡同时算不同列**。

```
TP=2：
GPU 0:  Y[:, 0:N/2]   = X × W[:, 0:N/2]
GPU 1:  Y[:, N/2:N]   = X × W[:, N/2:N]
```

每张卡只有 W 的一半，输出的 N 个维度被分到了两张卡。

---

## 2. Megatron-LM 的关键贡献

TP 的工程化方案来自 Megatron-LM 论文（NVIDIA, 2019）。

核心洞察：**不是所有矩阵都能直接切**。MLP 的两个矩阵 (fc1, fc2) 要配合切，否则要么结果错、要么通信量爆炸。

```
MLP: Y = GeLU(X × A) × B
     A: [K, N]    B: [N, K']

切法一（错）：A 切列、B 切列 → 中间结果要 all-gather（通信大）
切法二（对，Megatron）：A 切列、B 切行 → 中间结果不通信，最后 all-reduce
```

Megatron 的关键是 **Column + Row 配对切**：前一层列切，后一层行切，通信只发生在配对的边界。

---

## 3. TP 的两个切分方向

TP 切矩阵有两个方向，**取决于后续运算需要什么样的输入**：

| 切分 | 数学表达 | 切的是 | 通信需求 |
|------|---------|--------|---------|
| **Column Parallel** | A = [A_1, A_2, ..., A_p]（沿列切） | W 的输出维度 | forward 末尾 **all-gather**（如需完整输出） |
| **Row Parallel** | A = [A_1; A_2; ...; A_p]（沿行切） | W 的输入维度 | forward 末尾 **all-reduce**（每卡只算了部分和） |

详细数学见 [02_Linear层TP切分.md](02_Linear层TP切分.md)。

---

## 4. TP 的标准 MLP 切分

一个 MLP 块（fc1: K→4K，fc2: 4K→K）在 TP=N 下的切分：

```
输入 X: [M, K]  （每张卡都有完整副本）

GPU 0                            GPU 1
─────                            ─────
fc1: A[:, 0:2K]                  fc1: A[:, 2K:4K]
h = GeLU(X × A_0)  [M, 2K]       h = GeLU(X × A_1)  [M, 2K]
   （h 是局部的，不需要通信）       （h 是局部的，不需要通信）

fc2: B[0:2K, :]                  fc2: B[2K:4K, :]
Y_partial = h × B_0  [M, K]      Y_partial = h × B_1  [M, K]

Y = all_reduce(Y_partial_0, Y_partial_1)
                     ↑
              这是 TP 的唯一通信点
```

**关键**：

- fc1 Column Parallel + fc2 Row Parallel，通信只在 fc2 后 all-reduce
- fc1 后的激活 `h` 是**分片的**，不需要 all-gather（这是 Megatron 设计的精髓）

---

## 5. TP 的通信模式

### 5.1 All-reduce 是 TP 的命脉

TP 下，每个 TP rank 都需要与其他所有 rank 同步。通信原语是 **all-reduce**：

```
all-reduce：
  每张卡有一个分片张量 Y_i
  结果：每张卡都得到 sum(Y_i)
```

通信量与张量大小成正比：O(M×K×dtype_size)。

### 5.2 通信频率

TP 在 **每个需要 all-reduce 的层之后**都要做一次。对 Transformer：

- 每个 MLP 块的 fc2 之后（all-reduce）
- 每个 Attention 的 dense 之后（all-reduce）
- 如果 embedding 是 Column Parallel 且需要 gather_output，那里也要通信

**对比 PP**：PP 只在 stage 边界 send/recv（频率低）。
**对比 DP（训练）**：DP 在 backward 时 all-reduce 梯度（每个 step 一次）。

TP 的通信频率最高 → **必须要求低延迟高带宽互联**（NVLink/NVSwitch）。

---

## 6. TP 的代价与收益

### 6.1 代价

| 代价 | 表现 |
|------|------|
| 高频通信 | 每层 all-reduce，依赖 NVLink |
| 不跨节点 | 跨节点带宽不够，TP 性能崩塌 |
| 负载均衡 | 各 rank 计算量必须严格均等 |

### 6.2 收益

| 收益 | 原因 |
|------|------|
| 显存线性减少 | 权重按 N 倍切分到 N 张卡 |
| 单层算力线性提升 | N 张卡并行算同一个矩阵 |
| 不引入气泡 | 不像 PP 有空闲时间 |
| 实现简单 | 比起 PP 调度、EP 路由简单得多 |

---

## 7. TP 的适用边界

| 场景 | TP 是否合适 |
|------|------------|
| 单节点多卡（NVLink） | **合适**，主推 |
| 跨节点（普通以太网） | **不合适**，all-reduce 延迟太大 |
| 模型单层权重就很大 | **合适**（如 70B 模型，单层 30GB+） |
| 模型很浅但很宽 | **合适**（层少，PP 没意义） |
| GQA / MQA 模型 | **合适**，但要处理 KV head 不能整除的情况 |

**经验法则**：TP 通常配在 NVLink domain 内，TP size = 单节点 GPU 数（如 8）。

---

## 8. 与其他并行的边界

| 维度 | TP | PP | EP | DP |
|------|----|----|----|----|
| 切什么 | 矩阵列/行 | 层 | 专家 | batch |
| 通信 | all-reduce | send/recv | all-to-all | 训练 all-reduce |
| 频率 | 每层 | stage 边界 | MoE 层 | 每 step |
| 互联要求 | 高（NVLink） | 低 | 中 | 低 |
| 跨节点 | 不推荐 | 推荐 | 推荐 | 推荐 |

TP 是单节点内**降低单卡显存压力、加速单层算力**的主力。

---

## 9. 小结

- TP 本质：**矩阵按列或按行切到多张卡，最后 all-reduce 合并**
- Megatron 的精髓：Column + Row 配对切，把通信收敛到一次 all-reduce
- 通信模式：**每层 all-reduce**，依赖 NVLink，不跨节点
- 适用：单节点多卡、模型大、层数浅
