# 知识点 01：DP 基本原理

> 核心问题：DP 是什么？为什么需要它？跟 TP/PP/EP 怎么分工？

---

## 1. DP 的物理类比

把模型想象成一个工厂，DP 就是**开多个一模一样的工厂**：

```
工厂 A: 处理订单 1, 2, 3
工厂 B: 处理订单 4, 5, 6
工厂 C: 处理订单 7, 8, 9
```

- 每个工厂（DP rank）有完整的生产线（完整模型副本）
- 订单（请求）按工厂分配
- 工厂之间不共享半成品（除训练需要同步梯度）

---

## 2. DP 的本质

DP = **Data Parallelism（数据并行）**

切的是 **batch / 请求**，不切模型。

```
8 个请求 + 2 个 DP rank:
   DP rank 0: 处理请求 [0, 2, 4, 6]
   DP rank 1: 处理请求 [1, 3, 5, 7]
```

每张卡都持有**完整模型**，处理一部分数据。

---

## 3. DP 在训练和推理的差异

### 3.1 训练场景

```
每个 step:
   1. 每个 DP rank 处理一部分 batch（前向）
   2. 算 loss
   3. backward（反向传播）
   4. all-reduce 梯度（DP rank 间通信）
   5. 优化器更新参数

通信：每 step 一次 all-reduce（梯度）
```

### 3.2 推理场景

```
每个请求:
   1. 路由到某个 DP rank
   2. 该 rank 用完整模型 forward
   3. 返回结果

通信：**无**（DP rank 间不需要通信）
```

**关键**：推理 DP 几乎不通信，所以可以无限水平扩展。

---

## 4. DP 的适用条件

### 4.1 模型能单卡（或单 TP 组）放下

DP 要求每个 rank 持有完整模型。如果模型大，需要先 TP/PP 切：

```
Llama 70B（单卡 80GB 放不下）:
   单 DP rank = 一个 TP=8 组（占满一个节点）
   DP=4 → 4 个节点

DeepSeek V4 671B（更大）:
   单 DP rank = 一个 TP×PP 组（占满多节点）
   DP=M → M 个副本
```

### 4.2 吞吐受限（throughput-bound）

如果模型算力足够（throughput-bound 而非 memory-bound），加 DP rank 直接线性提升吞吐。

### 4.3 请求可独立处理

推理请求之间通常独立（不依赖彼此状态），适合 DP 切分。

---

## 5. DP vs TP/PP/EP 的本质区别

| 维度 | TP | PP | EP | **DP** |
|------|----|----|----|--------|
| 切什么 | 矩阵 | 层 | 专家 | **batch** |
| 模型是否完整 | 否 | 否 | 否 | **是（完整副本）** |
| 通信 | all-reduce | send/recv | all-to-all | 训练 all-reduce，推理无 |
| 显存节省 | 是 | 是 | 是 | **否**（每 rank 都有完整模型） |
| 目的 | 加速单层 | 切深模型 | 切 MoE | **扩展吞吐** |

**关键**：DP 是**唯一不切模型**的并行方式。

---

## 6. DP 的代价

### 6.1 显存不省

每个 DP rank 都持有完整模型，总显存 = N × 单模型。

例：Llama 70B 单副本 140GB（含 KV cache），DP=4 → 总显存 560GB。

### 6.2 模型加载慢

每个 rank 都要加载完整模型权重，加载时间不变（不像 TP/PP 加载 1/N）。

### 6.3 调度复杂

请求路由策略：哪个请求发到哪个 rank？需要负载均衡。

---

## 7. DP 的收益

| 收益 | 原因 |
|------|------|
| 线性扩展吞吐 | N 个副本 → N 倍吞吐（在请求充足的条件下） |
| 不引入气泡 | 不像 PP 有空闲时间 |
| 不增加单请求延迟 | 单请求仍在单副本上跑 |
| 实现简单 | 不需要切模型，只需要路由 |
| 容错好 | 一个副本挂了不影响其他 |

---

## 8. 跟 vLLM 主线设计的关系

vLLM 是推理框架，**DP 是它的吞吐扩展主力**。

```
单副本 vLLM 实例：
   - 一个 scheduler + N 个 worker
   - 处理一批请求

DP=N 的 vLLM：
   - N 个 scheduler + N × worker 组
   - 每组处理一部分请求
```

vLLM 通过 `--data-parallel-size N` 启动 DP 模式。

---

## 9. DP 的几种实现模式

### 9.1 多进程模式（vLLM 默认）

启动 N 个 vLLM 进程，每个是一个 DP rank：

```
vllm serve ... --data-parallel-size 4
   → 启动 4 个 vLLM 进程
   → 每个进程有自己的 scheduler + workers
   → 4 个进程共享 GPU（如果同节点）
```

通过环境变量 `VLLM_DP_RANK`、`VLLM_DP_WORLD_SIZE` 区分。

### 9.2 多容器模式（K8s）

K8s 部署，每个 pod 是一个 DP rank：

```
Pod 1: vLLM 实例 1
Pod 2: vLLM 实例 2
Pod 3: vLLM 实例 3
...

外部 LB（如 nginx、Kubernetes Service）路由请求
```

### 9.3 SPMD 模式（vLLM V1 探索）

同一进程跑多个 DP rank，通过 async scheduling 切换：

- 单进程内存共享，省内存
- 但实现复杂，性能不一定好

详见 [../L3-vLLM实现/04_DP架构与启动.md](../L3-vLLM实现/04_DP架构与启动.md)。

---

## 10. DP 与 EP 在 MoE 中的微妙关系

源码：`vllm/config/parallel.py:127-128`

```python
data_parallel_size: int = Field(default=1, ge=1)
"""Number of data parallel groups. MoE layers will be sharded according to
the product of the tensor parallel size and data parallel size."""
```

注释说：MoE 层按 **TP × DP** 切分。

这意味着 MoE 模型下，DP 增加 = EP 增加（expert 跨 DP rank 切分）。详见 [../L4-进阶场景/06_DP+EP_wide.md](../L4-进阶场景/06_DP+EP_wide.md)。

---

## 11. 小结

- DP 本质：**完整模型副本 + 切 batch**
- 推理 DP 几乎不通信，可水平扩展
- 显存不省（每 rank 都有完整模型）
- 是 vLLM 扩展吞吐的主力
- MoE 下 DP 会"借道"成为 EP
