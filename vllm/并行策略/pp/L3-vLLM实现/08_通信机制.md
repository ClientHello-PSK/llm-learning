# 知识点 08：PP 通信机制

> 核心问题：stage 间到底怎么 send/recv？HCCL/NCCL 怎么工作？

---

## 1. PP 通信的本质

PP 是**点对点通信**（send/recv），跟 TP 的 all-reduce 形成对比。

| 通信类型 | PP | TP | EP |
|---------|----|----|----|
| 模式 | 点对点 | 集体通信 | 集体通信 |
| 频率 | stage 边界 | 每层 | MoE 层 |
| 操作 | send_tensor_dict / recv_tensor_dict | all-reduce | all-to-all |
| 带宽要求 | 低 | 高 | 中 |

---

## 2. send/recv 的调用链

vLLM 中，PP 通信从高层到底层的调用链：

```
model.forward() 返回 IntermediateTensors
    ↓
GroupCoordinator.send_tensor_dict()
    ↓
DeviceCommunicator.send_tensor_dict()
    ↓
torch.distributed.send() / isend()
    ↓
底层后端
    ├─ GPU: NCCL
    └─ NPU: HCCL
```

### 关键 API（实际名称）

源码：`vllm/distributed/parallel_state.py`

```python
# GroupCoordinator 类的方法
def send_tensor_dict(self, tensor_dict, ...):     # L926 同步发送
def isend_tensor_dict(self, tensor_dict, ...):    # L964 异步发送
def recv_tensor_dict(self, ...):                  # L1021 同步接收
def irecv_tensor_dict(self, ...):                 # L1059 异步接收
```

> **注意**：vLLM **没有** `send_forward/recv_forward/send_backward/recv_backward` 这些方法。这是常见的误解。

## 3. NCCL vs HCCL

| 维度 | NCCL | HCCL |
|------|------|------|
| 全称 | NVIDIA Collective Communication Library | Huawei Collective Communication Library |
| 平台 | NVIDIA GPU | 华为 Ascend NPU |
| API | 类 MPI | 兼容 NCCL 风格 |
| 通信后端 | NVLink / InfiniBand / RoCE | HCCS / RoCE |
| 在 vLLM 中 | `NcclDeviceCommunicator` | `HcclDeviceCommunicator` |

vLLM 通过 `DeviceCommunicator` 抽象层屏蔽差异：

```python
# GroupCoordinator.__init__ (L380+)
if use_device_communicator and self.world_size > 1:
    device_comm_cls = resolve_obj_by_qualname(
        current_platform.get_device_communicator_cls()
    )
    self.device_communicator = device_comm_cls(...)
```

平台自动选择对应的 DeviceCommunicator 实现。

## 4. send_tensor_dict / recv_tensor_dict 的内部

### 4.1 send_tensor_dict（同步发送）

简化版逻辑：

```python
def send_tensor_dict(self, tensor_dict, dst_rank=None, ...):
    """发送 tensor_dict 给下一个 stage"""
    if dst_rank is None:
        dst_rank = self.next_rank  # 默认下一个 PP rank
    
    # 序列化 + 实际发送
    handles = self.isend_tensor_dict(tensor_dict, dst_rank, ...)
    # 等待完成
    for handle in handles:
        handle.wait()
```

### 4.2 recv_tensor_dict（同步接收）

简化版逻辑：

```python
def recv_tensor_dict(self, src_rank=None, ...):
    """从上一个 stage 接收 tensor_dict"""
    if src_rank is None:
        src_rank = self.prev_rank  # 默认上一个 PP rank
    
    # 准备接收 buffer + 实际接收
    tensor_dict, handles, _ = self.irecv_tensor_dict(...)
    for handle in handles:
        handle.wait()
    return tensor_dict
```

### 4.3 异步版本（isend/irecv）

异步版本返回 handles，不阻塞调用方：

```python
handles = pp_group.isend_tensor_dict(tensor_dict, dst_rank)
# 不等发送完成，继续做其他计算
...
for handle in handles:
    handle.wait()  # 显式等待
```

## 5. tensor_dict 的序列化

### 5.1 为什么要序列化？

`IntermediateTensors` 是 dict 容器，可能含多个 tensor。底层 `torch.distributed.send` 一次只能发一个连续 buffer。

### 5.2 实现方式

vLLM 内部把 dict 序列化成连续 buffer 发送，接收端反序列化还原：

```
发送端：dict → metadata + tensor buffers → send
接收端：recv → tensor buffers + metadata → dict
```

metadata 包含每个 tensor 的 shape、dtype、name 等信息。

### 5.3 关键约束

- **dtype 一致**：发送和接收的 tensor dtype 必须匹配
- **shape 已知**：接收端需要预先知道 shape（通过 metadata）
- **顺序固定**：发送/接收必须配对

## 6. 异步通信：isend / irecv 的价值

### 6.1 同步通信的问题

```
同步通信：
  |计算 stage0|等 send|recv|计算 stage1|
                            ↑通信时 GPU/NPU 空闲
```

### 6.2 异步通信的方案

```
异步通信：
  |计算 stage0|isend|开始计算下一批|
                ↓
            后台传输
                ↓
            下一个 step 时 irecv 已经完成
```

### 6.3 跟 async-scheduling 的关系

异步通信是 async-scheduling 的基础。详见 [L4/10_PP+async_scheduling.md](../L4-进阶场景/10_PP+async_scheduling.md)。

## 7. group 概念

### 7.1 device_group / cpu_group

GroupCoordinator 维护两个通信组：

```python
self.cpu_group = ...    # CPU 通信组（gloo 后端，用于元数据协调）
self.device_group = ...  # 设备通信组（NCCL/HCCL，用于张量传输）
```

所有 send_tensor_dict / broadcast 都在 device_group 上进行。

### 7.2 跟 TP/EP 的 group 区分

vLLM 中每种并行有独立的 group，但都是 `GroupCoordinator` 实例：

```python
_PP = GroupCoordinator(...)  # PP 通信组
_TP = GroupCoordinator(...)  # TP 通信组
_EP = GroupCoordinator(...)  # EP 通信组
```

通过 `get_pp_group() / get_tp_group() / get_ep_group()` 获取。

group 之间互不干扰。

## 8. PP 通信的常见错误

### 8.1 Shape mismatch

```python
# 上游 stage
return IntermediateTensors({"hidden_states": torch.zeros(10, 100)})

# 下游 stage
hidden_states = intermediate_tensors["hidden_states"]  # 期望 (10, 100)
# 但上游改成 (10, 200) 了 → shape mismatch → send/recv 失败
```

**解决**：修改 PP 间传递的张量 shape 时，必须同步上下游代码。

### 8.2 dtype mismatch

```python
# 上游是 fp16
# 下游期望 bf16
# → 数据解释错误
```

**解决**：PP 间张量统一 dtype（通常 bf16）。

### 8.3 死锁

```python
# stage 0 send
# stage 1 send（不 recv）
# → 双方都在等 → 死锁
```

**解决**：保证 send/recv 配对（一个 send 必有对应 recv）。

### 8.4 使用不存在的 API

```python
# ❌ 这些方法都不存在
pp_group.send_forward(...)
pp_group.recv_forward(...)
pp_group.send_backward(...)
pp_group.recv_backward(...)

# ✓ 正确 API
pp_group.send_tensor_dict(...)
pp_group.recv_tensor_dict(...)
pp_group.isend_tensor_dict(...)
pp_group.irecv_tensor_dict(...)
```

## 9. 推理场景的简化

推理**只有 forward**，所以：

- 只用 `send_tensor_dict` / `recv_tensor_dict`（forward 方向）
- 不需要反向通信

但 **PP+MTP 场景**需要反向通信（last rank 广播 token 给其他 rank），用 `broadcast`：

```python
# last rank 把采样结果广播给所有 PP rank
# 注意：src 用 last_rank（返回全局 rank），不是 is_last_rank（返回 bool）
dist.broadcast(
    sampled_token_ids,
    src=pp_group.last_rank,  # 或 pp_group.ranks[-1]
    group=pp_group.device_group
)
```

详见 [L4/09_PP+MTP.md](../L4-进阶场景/09_PP+MTP.md)。

## 10. 性能考量

### 10.1 通信量

每次 PP 通信量 = activation 大小 = `num_tokens × hidden_dim × dtype_size`

**例**：1000 tokens, hidden_dim=8192, bf16（2 bytes）
```
通信量 = 1000 × 8192 × 2 = 16 MB
```

### 10.2 通信延迟

PP 通信延迟包括：
- 序列化 / 反序列化
- 网络传输（依赖互联带宽）
- 同步等待

跟 TP 的 all-reduce 比，PP 的通信延迟通常小一个数量级。

### 10.3 优化手段

| 手段 | 效果 |
|------|------|
| 异步通信（isend_tensor_dict/irecv_tensor_dict） | 隐藏通信延迟 |
| 张量融合（多 tensor 一起发送） | 减少通信次数 |
| Async scheduling | 跨 step 隐藏通信 |
| 高速互联（NVLink/HCCS） | 提升带宽 |

## 12. 小结

- PP 通信是**点对点 send/recv**（vs TP 的 all-reduce）
- API 是 `send_tensor_dict / recv_tensor_dict`（不是 send_forward/recv_forward）
- 调用链：`send_tensor_dict` → `device_communicator.send_tensor_dict` → `torch.distributed.send` → NCCL/HCCL
- IntermediateTensors 内部 dict 通过序列化 + metadata 传输
- 异步通信（isend/irecv）是隐藏延迟的关键
- 推理只 forward 方向，PP+MTP 需要反向 broadcast
