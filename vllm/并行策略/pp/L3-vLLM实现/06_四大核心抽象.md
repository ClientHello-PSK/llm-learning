# 知识点 06：vLLM PP 的核心抽象

> 核心问题：vLLM 实现 PP 用了哪些核心类？它们各自负责什么？
> 源码位置：`vllm/distributed/parallel_state.py`、`vllm/sequence.py`

---

## 1. 核心抽象总览

| 抽象 | 类型 | 职责 |
|------|------|------|
| `GroupCoordinator` | 通用进程组 | **统一管理**所有并行组（PP/TP/EP/DP），提供 rank 查询、send/recv 通信 |
| `IntermediateTensors` | dataclass 容器 | PP stage 间传递张量的载体 |
| `get_pp_group()` | 全局入口 | 获取当前进程的 PP 组实例（返回 `GroupCoordinator`） |
| `DeviceCommunicator` | 设备通信器 | 底层 NCCL/HCCL 抽象，被 GroupCoordinator 组合使用 |

**重要**：vLLM **没有专门的 PP 类**（如 `PipelineParallelGroup` 或 `PipelineParallelCoordinator`）。所有并行组都用同一个 `GroupCoordinator` 类，通过不同的全局变量（`_TP`、`_PP`、`_EP`、`_DP`）区分。

---

## 2. GroupCoordinator

### 2.1 职责

源码：`vllm/distributed/parallel_state.py:351`

`GroupCoordinator` 是 vLLM 所有并行组的统一基类，负责：

- 维护进程组（cpu_group + device_group）
- 提供 rank 查询接口
- 提供 send/recv/broadcast 通信接口（通过 `device_communicator`）

### 2.2 关键属性（L361-378）

```python
class GroupCoordinator:
    # available attributes:
    rank: int                 # 全局 rank
    ranks: list[int]          # 组内所有全局 rank 列表
    world_size: int           # 组大小
    local_rank: int           # 节点内 local rank
    rank_in_group: int        # 组内 rank
    cpu_group: ProcessGroup   # CPU 通信组（gloo）
    device_group: ProcessGroup  # 设备通信组（NCCL/HCCL）
    device_communicator: DeviceCommunicatorBase | None  # 设备通信器
```

### 2.3 Rank 查询接口（L530-557）

```python
@property
def first_rank(self):
    """组内第一个进程的全局 rank"""
    return self.ranks[0]

@property
def last_rank(self):
    """组内最后一个进程的全局 rank"""
    return self.ranks[-1]

@property
def is_first_rank(self):
    """当前进程是否是组内第一个"""
    return self.rank == self.first_rank

@property
def is_last_rank(self):
    """当前进程是否是组内最后一个"""
    return self.rank == self.last_rank

@property
def next_rank(self):
    """下一个进程的全局 rank"""
    rank_in_group = self.rank_in_group
    return self.ranks[(rank_in_group + 1) % self.world_size]
```

### 2.4 关键认知

- `first_rank`、`last_rank` 是 **property**，返回**全局 rank**（不是 bool）
- `is_first_rank`、`is_last_rank` 是 **property**，返回 **bool**
- 当用作 `dist.broadcast(src=...)` 时，应该用 `last_rank`（返回全局 rank）
- 当用作 `if pp.is_last_rank:` 时，应该用 `is_last_rank`

### 2.5 容易混淆的 API

| 用途 | 正确 API | 返回 |
|------|---------|------|
| 判断是否首个 | `pp.is_first_rank` | bool |
| 判断是否末个 | `pp.is_last_rank` | bool |
| 获取首个全局 rank | `pp.first_rank` | int |
| 获取末个全局 rank | `pp.last_rank` | int |
| 获取所有 rank | `pp.ranks` | list[int] |
| 获取组大小 | `pp.world_size` | int |

## 3. send/recv 通信 API

### 3.1 实际方法名

**vLLM 用 `send_tensor_dict / recv_tensor_dict`**，不是 `send_forward / recv_forward`。

源码 `vllm/distributed/parallel_state.py`（在 GroupCoordinator 内）：

```python
def send_tensor_dict(self, tensor_dict, ...):       # L926 同步发送
def isend_tensor_dict(self, tensor_dict, ...):      # L964 异步发送
def recv_tensor_dict(self, ...):                    # L1021 同步接收
def irecv_tensor_dict(self, ...):                   # L1059 异步接收
```

### 3.2 推理场景的实际调用

```python
# 上游 stage 发送
pp_group.send_tensor_dict({"hidden_states": hidden_states})

# 下游 stage 接收
tensor_dict = pp_group.recv_tensor_dict()
hidden_states = tensor_dict["hidden_states"]
```

### 3.3 跟 `IntermediateTensors` 的关系

`IntermediateTensors` 是 model.forward 层面的抽象；底层通信时，`GroupCoordinator.send_tensor_dict` 直接传 dict，不强制用 `IntermediateTensors`。

## 4. IntermediateTensors

### 4.1 实际定义

源码：`vllm/sequence.py:12`

```python
@dataclass
class IntermediateTensors:
    """For all pipeline stages except the last, we need to return the hidden
    states and residuals to be sent to the next stage."""
    
    tensors: dict[str, torch.Tensor]
    kv_connector_output: KVConnectorOutput | None
    
    def __init__(self, tensors, kv_connector_output=None):
        self.tensors = tensors
        self.kv_connector_output = kv_connector_output
    
    def __getitem__(self, key):
        if isinstance(key, str):
            return self.tensors[key]
        elif isinstance(key, slice):
            return self.__class__({k: v[key] for k, v in self.tensors.items()})
```

### 4.2 关键事实

- 是 **`@dataclass`**，不是 `UserDict` 子类
- 字段是 `tensors: dict`，不是直接继承 dict
- 通过 `__getitem__` 支持 `tensors["hidden_states"]` 访问
- 额外有 `kv_connector_output` 字段（KV connector 相关）

### 4.3 为什么是 dict？

跟之前文档说法一致：

1. **多种张量**：可能传 hidden_states、metadata、attention_metadata 等
2. **灵活性**：不同模型可传不同字段
3. **键控访问**：通过 key 区分

### 4.4 典型用法

**上游 stage**（DeepseekV4Model.forward L1159-1164）：

```python
if not get_pp_group().is_last_rank:
    return IntermediateTensors({
        "hidden_states": hidden_states,
    })
```

**下游 stage**（L1117-1118）：

```python
assert intermediate_tensors is not None
hidden_states = intermediate_tensors["hidden_states"]
```

## 5. get_pp_group()

### 5.1 实际定义

源码：`vllm/distributed/parallel_state.py:1347-1352`

```python
_PP: GroupCoordinator | None = None

def get_pp_group() -> GroupCoordinator:
    assert _PP is not None, "pipeline model parallel group is not initialized"
    return _PP
```

**返回类型是 `GroupCoordinator`**（不是某个 PP 专用类）。

### 5.2 全局并行组变量

vLLM 用全局变量保存每种并行组：

```python
_TP: GroupCoordinator | None = None    # L1331
_PP: GroupCoordinator | None = None    # L1347
_DP: GroupCoordinator | None = None    # L1355
_EP: GroupCoordinator | None = None    # L1363
```

每种并行都是同一个类 `GroupCoordinator`，只是包含的 ranks 不同。

### 5.3 相关 API

| API | 用途 |
|-----|------|
| `get_pp_group()` | 获取 PP 组 |
| `get_tp_group()` | 获取 TP 组 |
| `get_ep_group()` | 获取 EP 组 |
| `get_dp_group()` | 获取 DP 组 |

## 6. DeviceCommunicator（底层通信）

### 6.1 职责

`GroupCoordinator` 内部组合了 `DeviceCommunicator`，负责底层 NCCL/HCCL 抽象。

### 6.2 平台差异

| 平台 | DeviceCommunicator 实现 |
|------|------------------------|
| NVIDIA GPU | `NcclDeviceCommunicator`（基于 NCCL） |
| 华为 Ascend NPU | `HcclDeviceCommunicator`（基于 HCCL） |
| CPU | `CpuCommunicator`（基于 gloo） |

vLLM 通过 `current_platform.get_device_communicator_cls()` 动态选择。

### 6.3 调用链

```
GroupCoordinator.send_tensor_dict()
    ↓
self.device_communicator.send_tensor_dict()  # 实际通信
    ↓
torch.distributed.send() / isend()
    ↓
NCCL / HCCL
```

## 7. 核心抽象的协作关系

```
vLLM 启动
    ↓
initialize_model_parallel()
    ↓
创建多个 GroupCoordinator 实例（PP/TP/EP/DP）
    ↓
存入全局 _PP / _TP / _EP / _DP
    ↓
ModelRunner.run()
    ↓
model.forward()
    ├─ get_pp_group() 获取 PP 组
    ├─ 判断 is_first_rank / is_last_rank
    ├─ 不 first rank: 从 IntermediateTensors 读 hidden_states
    ├─ 跑 layers[start:end]
    ├─ 不 last rank: 返回 IntermediateTensors({"hidden_states": ...})
    │       ↓
    │   上层调用 pp_group.send_tensor_dict() 实际传输
    └─ last rank: 返回最终 hidden_states
```

## 9. 常见误用提醒（修正版）

| 误用 | 正确 |
|------|------|
| `class PipelineParallelGroup`（不存在） | 用 `GroupCoordinator` |
| `pp.send_forward()`（不存在） | 用 `pp.send_tensor_dict()` |
| `pp.recv_forward()`（不存在） | 用 `pp.recv_tensor_dict()` |
| `IntermediateTensors` 在 `communication_op.py` | 在 `vllm/sequence.py` |
| `IntermediateTensors` 继承 UserDict | 是 `@dataclass` |
| `get_pp_group` 未 import | `from vllm.distributed.parallel_state import get_pp_group` |

**注意**：之前 PR review 中提到的 `pp.last_rank` 不存在的问题，**现在已不成立**——`last_rank` 是 `GroupCoordinator` 的 property（L535）。早期的 vLLM 版本可能没有，但当前主分支已存在。

---

## 10. 小结（修正版）

- vLLM **没有专门的 PP 类**，统一用 `GroupCoordinator` 管理 PP/TP/EP/DP
- 通信 API 是 `send_tensor_dict / recv_tensor_dict`（不是 send_forward/recv_forward）
- `IntermediateTensors` 是 dataclass，定义在 `vllm/sequence.py:12`
- `get_pp_group()` 返回 `GroupCoordinator` 实例
- `is_first_rank/is_last_rank` 是 bool property；`first_rank/last_rank` 是 int property
