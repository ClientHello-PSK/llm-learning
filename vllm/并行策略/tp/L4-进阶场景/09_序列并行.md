# 知识点 09：Sequence Parallel (SP)

> 核心问题：什么是序列并行？它跟 TP 怎么配合？vLLM 怎么实现？

---

## 1. SP 的动机

### 1.1 标准 TP 的问题

标准 TP 下：

```
LayerNorm 之前，激活是完整的（每 rank 都有 [M, K]）
LayerNorm 计算：每 rank 都做一遍（重复计算）
```

当序列很长（M 大）时：

- LayerNorm 的激活内存 = M × K × dtype × num_ranks（重复存储）
- LayerNorm 计算重复 num_ranks 倍

### 1.2 SP 的解决思路

**SP：把 sequence 维（M）切到 TP rank**

```
原本: [M, K] 在每 rank 完整
SP:   [M/p, K] 在每 rank（每 rank 拿 1/p 的 token）
```

激活内存降低 p 倍，计算量也降低 p 倍。

---

## 2. SP 的数学逻辑

### 2.1 LayerNorm 在 SP 下

LayerNorm 沿 hidden_dim（K）做归一化，跟 sequence（M）独立：

```
原 LayerNorm(x): x 是 [M, K]，每 token 独立归一化
SP LayerNorm(x): x 是 [M/p, K]，每 rank 算自己 1/p 的 token

数学等价！
```

### 2.2 Attention 在 SP 下

Attention 跨 token 交互，SP 切 sequence 后，Q/K/V 还需要拼回完整 sequence：

```
SP 状态: Q_i: [M/p, head_dim]  每 rank 1/p 的 Q

要做 attention，需要：
   Q_full = all_gather([Q_0, Q_1, ..., Q_{p-1}])  shape: [M, head_dim]
```

或者用 ring-attention（更复杂）。

### 2.3 MLP 在 SP 下

MLP 的 fc1 是 Column Parallel（切 N）：

```
标准 TP: Y = X × A_i  X 是 [M, K] 完整，Y_i: [M, N/p]
SP+TP:   Y = X_i × A_i  X_i 是 [M/p, K] 1/p，Y_i: [M/p, N/p]
```

输出更小，但需要后续处理才能 all-reduce。

### 2.4 输出后 reduce-scatter

MLP 末尾原本是 all-reduce，SP 下改为 **reduce-scatter**：

```
all-reduce:    [M, N/p] × p → [M, N/p]  (每 rank 完整)
reduce-scatter: [M, N/p] × p → [M/p, N/p] (每 rank 1/p，但 N 完整)

reduce-scatter 后又回到 SP 状态（[M/p, N/p]）
```

---

## 3. SP 的通信循环

```
1. LayerNorm (SP: [M/p, K])
2. before attention: all-gather sequence 维 → [M, K]（完整）
3. QKV proj (TP): [M, K] → [M, N_qkv/p]
4. attention: [M, N_qkv/p] → [M, N_o/p]
5. o_proj (TP): all-reduce → [M, K]
6. after o_proj: reduce-scatter sequence 维 → [M/p, K]（回到 SP）

回到 1（下一个 block）
```

**关键观察**：

- 每个 block 多了一次 all-gather 和一次 reduce-scatter
- 但 LayerNorm/RMSNorm 部分省了 p 倍内存
- 通信总量跟原 TP 差不多（all-reduce = reduce-scatter + all-gather）

## 4. vLLM 的 SP 实现

### 4.1 vLLM 何时用 SP？

vLLM 的 SP 主要用于 **编译优化**（compilation passes）。

源码：`vllm/compilation/passes/fusion/sequence_parallelism.py`

通过融合 all-reduce + RMSNorm，减少通信次数。

### 4.2 融合示例：all-reduce + RMSNorm

```
传统：
   down_proj → output_parallel: [M, K/p]
   all-reduce → output: [M, K/p] (但 K 是完整维度)
   RMSNorm(output)

融合：
   all-reduce + RMSNorm 一起做
   通信的同时计算 norm

实现：vllm/compilation/passes/fusion/allreduce_rms_fusion.py
```

### 4.3 reduce-scatter 接口

源码：`vllm/distributed/communication_op.py:24`

```python
def tensor_model_parallel_reduce_scatter(input_, dim=-1):
    return get_tp_group().reduce_scatter(input_, dim)
```

SP 的关键通信原语。

---

## 5. SP 的代价

### 5.1 attention 的麻烦

Attention 跨 token 交互，SP 切 sequence 后，attention 计算要么：

- **all-gather Q/K/V**（标准做法）：通信开销大
- **ring-attention**：环式通信，分块计算，但实现复杂

### 5.2 不适合 prefill 长序列

prefill 时序列长，attention 是 O(M²) 复杂度，SP 切 sequence 后 ring-attention 才划算。

### 5.3 decode 阶段不需要 SP

decode 时 M=1（每 step 处理一个 token），SP 切 sequence 没意义。

vLLM 主要在 prefill 阶段考虑 SP 优化。

---

## 6. SP 跟 Context Parallel (CP) 的区别

| 维度 | SP | CP |
|------|----|----|
| 切什么 | LayerNorm 前的激活 | KV cache 中的 sequence |
| 切哪里 | LayerNorm 路径 | attention 的 sequence 维 |
| 通信 | all-gather / reduce-scatter | ring-attention |
| vLLM 配置 | 默认编译优化 | `prefill_context_parallel_size` |
| 适用 | 跟 TP 配合减少激活内存 | 跨节点 prefill 长序列 |

CP 跟 SP 都切 sequence，但 SP 是 TP 内部优化，CP 是另一个独立的并行维度。

---

## 7. ULTRA-SP / DeepSpeed Ulysses

更激进的 SP 方案：

```
1. Q/K/V 按 head 切（TP）
2. 重排：head 维跟 sequence 维交换（用 all-to-all）
3. 每 rank 拿一部分 head 的完整 sequence
4. attention 用自己 head 段的完整 sequence
5. all-to-all 切回
```

适合 head 多、序列长的场景。

vLLM 暂未广泛采用 Ulysses，是研究方向。

---

## 8. vLLM 中 SP 的现状

### 8.1 主要场景

- **all-reduce + RMSNorm 融合**：减少通信延迟
- **attention_metadata 的 sequence 切分**：用于编译优化

### 8.2 配置

vLLM 没有独立的 SP 配置开关，SP 通过编译 pass 自动启用。

```python
# vllm/compilation/passes/fusion/sequence_parallelism.py
# 自动检测 RMSNorm 后的 all-reduce 模式，融合成 SP 形式
```

### 8.3 性能

对长序列 prefill 有 5-10% 性能提升，对 decode 影响小。

## 10. 小结

- SP：把 sequence 维切到 TP rank，减少 LayerNorm 激活内存
- 通信循环：LayerNorm (SP) → all-gather → attention/MLP → reduce-scatter → LayerNorm (SP)
- vLLM 用 SP 做编译融合（all-reduce + RMSNorm），不强制使用
- 跟 Context Parallel (CP) 区别：SP 在 TP 内部，CP 是独立维度
