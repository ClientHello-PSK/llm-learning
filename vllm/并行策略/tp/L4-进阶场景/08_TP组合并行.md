# 知识点 08：TP 与其他并行的组合

> 核心问题：TP 跟 PP/EP/DP 怎么协同？模型 layout 是什么？

---

## 1. 多维并行的 layout

源码：`vllm/distributed/parallel_state.py:1740`

```
layout 顺序: ExternalDP × DP × PP × PCP × TP
```

含义：world_size 个 rank 被组织成一个多维数组，从外到内是：

```
最外层: ExternalDP（外部 DP，不在 vLLM 内）
       ↓
       DP（vLLM 内部 DP）
       ↓
       PP（流水线 stage）
       ↓
       PCP（prefill context parallel）
       ↓
最内层: TP（tensor parallel，最内层意味着相邻 rank）
```

### 1.1 举例：8 卡、TP=2、PP=2、DP=2

```
world_size = 8
all_ranks.reshape(2, 2, 1, 1, 2):
                          ↑  ↑  ↑  ↑  ↑
                       ExternalDP DP PP PCP TP

rank 编号（从 0 开始）：
DP rank 0:
    PP rank 0:
        TP rank 0: global rank 0
        TP rank 1: global rank 1
    PP rank 1:
        TP rank 0: global rank 2
        TP rank 1: global rank 3

DP rank 1:
    PP rank 0:
        TP rank 0: global rank 4
        TP rank 1: global rank 5
    PP rank 1:
        TP rank 0: global rank 6
        TP rank 1: global rank 7
```

### 1.2 同 TP 组的 rank 是相邻的

因为 TP 在最内层，同一 TP 组的 ranks 在物理上相邻（同节点内），符合"TP 不跨节点"的要求。

## 2. TP × PP 组合

### 2.1 拓扑结构

```
TP × PP 模型 layout：

TP=2, PP=2 (4 卡):

   TP rank 0      TP rank 1
  ┌──────────┐  ┌──────────┐
  │ Layer 0  │  │ Layer 0  │     ← PP stage 0
  │ ...      │  │ ...      │
  │ Layer 15 │  │ Layer 15 │
  └────┬─────┘  └────┬─────┘
       │             │
       └──────┬──────┘   PP 边界 (send/recv)
              │
  ┌──────────┐  ┌──────────┐
  │ Layer 16 │  │ Layer 16 │     ← PP stage 1
  │ ...      │  │ ...      │
  │ Layer 31 │  │ Layer 31 │
  └──────────┘  └──────────┘
```

### 2.2 通信模式

```
Stage 0 内部（TP）：每层 all-reduce
Stage 边界：PP send/recv

每张卡既参与 TP 通信（all-reduce），也参与 PP 通信（send/recv）
```

### 2.3 send/recv 的 TP 配合

PP stage 边界传 hidden_states：

- 上游 stage 的 **TP 组 all-reduce 之后** 的 hidden_states（完整）
- 下游 stage 的每个 TP rank 都接收完整 hidden_states

源码中 `IntermediateTensors` 容器承载 PP 间的张量，但实际传输的是 hidden_states（all-reduce 后的结果）。

### 2.4 为什么 TP 在 PP 之前？

数学等价性：TP 是单层内切，PP 是层间切，两者独立。

但工程上：

- **TP all-reduce 必须在 stage 之前完成**：因为 PP 下游需要完整的 hidden_states
- 否则下游 stage 收到的是分片，又要做 TP all-gather，浪费通信

## 3. TP × EP 组合（MoE）

### 3.1 MoE 模型的并行

MoE 模型：

- **Attention 部分**：跟 Dense 一样，用 TP（按 head 切）
- **Expert 部分**：用 EP（按 expert 切）

### 3.2 标准 MoE block

```
hidden: [M, K]
   ↓
attention (TP)
   ↓
   ↓ Router/Gating
   ↓
expert dispatch (all-to-all, EP)
   ↓
expert computation (每 expert 在自己 rank)
   ↓
expert combine (all-to-all, EP)
   ↓
hidden_out: [M, K]
```

### 3.3 TP 和 EP 是否同一个组？

不一定，取决于配置：

**Case A：TP=EP（同组）**

```
Attention TP 切 head，Expert EP 切 expert
两者用同一个 group，all-reduce / all-to-all 都在这组
```

**Case B：TP ≠ EP（不同组）**

```
Attention 用 TP 组（all-reduce）
Expert 用 EP 组（all-to-all）
两个 group 不同，ranks 不同

例：TP=2 (单节点), EP=8 (跨 4 节点 × 2 卡/节点)
```

### 3.4 layout 体现

源码：`vllm/distributed/parallel_state.py:1850-1876`

```python
# Build the EP group
# Don't create EP group for dense models.
if config.model_config is None or config.model_config.is_moe:
    group_ranks = (
        all_ranks.transpose(1, 2)  # 交换 DP 和 PP 维
        .reshape(
            -1,
            data_parallel_size
            * prefill_context_model_parallel_size
            * tensor_model_parallel_size,
        )
        .unbind(0)
    )
    ...
```

EP 组跨 DP、PP、TP 多维，可以跨节点。

## 4. TP × DP 组合

### 4.1 模型副本

DP 的本质是**多份完整模型副本**：

```
TP=2, DP=2 (4 卡):

DP rank 0 (model replica 0):
    TP rank 0: 卡 0
    TP rank 1: 卡 1
    → 这两张卡组成 model replica 0

DP rank 1 (model replica 1):
    TP rank 0: 卡 2
    TP rank 1: 卡 3
    → 这两张卡组成 model replica 1
```

### 4.2 通信模式

- **DP 组内几乎不通信**（推理场景）：每个副本独立处理请求
- **TP 组内 all-reduce**：每个副本内部
- **DP 组间不通信**：除非是 MoE 模型（expert 跨副本切分，详见 [../../dp/L4-进阶场景/06_DP+EP_wide.md](../../dp/L4-进阶场景/06_DP+EP_wide.md)）

### 4.3 MoE 下 DP 和 TP 共同切 expert

源码：`vllm/config/parallel.py:127-128`

```python
data_parallel_size: int = Field(default=1, ge=1)
"""Number of data parallel groups. MoE layers will be sharded according to
the product of the tensor parallel size and data parallel size."""
```

注释说：MoE 层按 `TP × DP` 切分。所以 wide-EP 场景下，DP 增加相当于 EP 增加。

## 5. TP × SP（Sequence Parallel）

### 5.1 SP 是什么

Sequence Parallel：把 sequence 维度（dim=1）切到 TP rank。

跟 TP 配合，把 **LayerNorm 后的输入** 按 sequence 切，减少激活内存。

### 5.2 与 TP 的关系

```
标准 TP：
   LayerNorm: 输入完整（每 rank 都有）
   QKV proj: 切 head
   Attention: 每 rank 自己 head
   O proj: all-reduce

SP + TP：
   LayerNorm: 输入按 sequence 切
   QKV proj: 之前要 all-gather 拼回完整 sequence（或用 reduce-scatter）
   ...
```

详见 [09_序列并行.md](09_序列并行.md)。

### 5.3 vLLM 的实现

vLLM 在 `vllm/compilation/passes/fusion/sequence_parallelism.py` 中实现 SP 融合，主要用于编译优化。

## 6. 选择组合的策略

### 6.1 决策树

```
单节点多卡（8 GPU）：
   ├─ Dense 小模型（< 30B）：TP=8
   ├─ Dense 大模型（70B+）：TP=8 或 TP=4 × PP=2
   └─ MoE 大模型：TP=2/4 × EP=剩余

跨节点：
   ├─ Dense 大模型：PP 跨节点 × TP 单节点
   └─ MoE：EP 跨节点 × TP 单节点
```

### 6.2 DeepSeek V4 671B 案例

```
PP × EP × TP × DP
 8   × N  ×  2 ×  M
```

- **PP=8**：跨 8 节点切层（每节点一个 stage）
- **EP=N**：每 stage 内 MoE expert 跨剩余卡切
- **TP=2**：节点内 2 卡 TP（attention 部分）
- **DP=M**：剩余卡做副本

### 6.3 选 TP 的关键约束

- **TP 通常 = 单节点 GPU 数**：因为 TP 不跨节点
- **TP 必须能整除 num_heads**：head 切分约束
- **TP 越大，all-reduce 越慢**：单层 GEMM 快，但 all-reduce 开销增大
- **TP=2 或 4 是常见选择**：TP=8 通常配合 PP 跨节点

## 7. 多维并行下的进程组数量

例：world_size=16, TP=2, PP=4, DP=2：

```
TP 组数：16 / 2 = 8
PP 组数：16 / 4 = 4
DP 组数：16 / 2 = 8

每张卡属于：
   1 个 TP 组
   1 个 PP 组
   1 个 DP 组
   （如果是 MoE，还有 1 个 EP 组）
```

每个进程组是独立的 NCCL communicator，内存开销随组数线性增加。

---

## 8. 小结

- layout 顺序：`ExternalDP × DP × PP × PCP × TP`（TP 最内层，保证同节点）
- TP × PP：标准大模型组合，TP all-reduce 在 stage 边界之前完成
- TP × EP：MoE 模型，attention 用 TP，expert 用 EP
- TP × DP：多副本，DP 组内几乎不通信
- TP × SP：序列并行，减少激活内存
