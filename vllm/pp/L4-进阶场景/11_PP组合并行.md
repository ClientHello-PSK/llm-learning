# 知识点 11：PP 组合并行（PP + TP + EP）

> 核心问题：实际部署时 PP 几乎不单独使用，怎么跟 TP、EP 组合？

---

## 1. 为什么需要组合并行？

单种并行有上限：

| 并行 | 上限 |
|------|------|
| TP | 单节点（NVLink 限制） |
| PP | stage 数过多会气泡爆炸 |
| EP | 只切 MoE 层，dense 层没分 |

**实际部署**：用多种并行组合，每种负责自己擅长的部分。

---

## 2. PP + TP（最常见的组合）

### 2.1 切分逻辑

```
模型 64 层，PP=2, TP=2

PP 切层（粗粒度）：
  PP Stage 0: layer 0-31
  PP Stage 1: layer 32-63

TP 切矩阵（细粒度，每个 PP stage 内部）：
  每个 PP stage 内部 TP=2，矩阵切成 2 块
```

### 2.2 通信模式

```
                    PP 通信（stage 间）
                    ↓
PP Stage 0 ──────────────────────── PP Stage 1
  │  TP 内部 all-reduce                │  TP 内部 all-reduce
  │  （每层）                          │  （每层）
  ↓                                    ↓
TP rank 0, 1                         TP rank 0, 1
```

- **stage 内**：每层 all-reduce（TP 通信）
- **stage 间**：边界 send/recv（PP 通信）

### 2.3 总通信量

每层 TP all-reduce + stage 边界 PP send/recv。

**例**：64 层，PP=2, TP=2
- PP 边界 send/recv：1 次（layer 31 → layer 32）
- TP all-reduce：64 次（每层）

TP 通信远多于 PP，所以 TP 适合节点内（高速互联），PP 适合节点间。

### 2.4 rank 分配

PP=2, TP=2，总共 4 个 rank：

```
TP Group 0: rank 0, rank 1（PP Stage 0）
TP Group 1: rank 2, rank 3（PP Stage 1）

PP Group 0: rank 0, rank 2（TP rank 0）
PP Group 1: rank 1, rank 3（TP rank 1）
```

每个 rank 同时属于一个 TP 组和一个 PP 组。

---

## 3. PP + EP（MoE 模型）

### 3.1 MoE 模型的特殊性

MoE 层有多个专家：

```
256 个专家，每次 token 路由到 top-8 个
```

dense 层和 attention 层跟普通模型一样。

### 3.2 EP 切专家

```
256 个专家，EP=8
每个 EP rank 放 32 个专家
token 路由到对应 rank
```

### 3.3 跟 PP 配合

```
PP 切 dense 层和 attention 层
EP 切 MoE 层的专家

PP Stage 0: dense_0, attn_0, MoE_0（部分专家）
PP Stage 1: dense_1, attn_1, MoE_1（部分专家）
```

### 3.4 通信复杂度

PP+EP 比 PP+TP 复杂：

- **PP 边界**：send/recv activation
- **EP 内部**：all-to-all 路由 token 到对应专家
- **MoE 层的特殊性**：每个 PP stage 内的 MoE 层都要做 EP 通信

---

## 4. PP + TP + EP（三维并行）

### 4.1 DeepSeek V4 的典型配置

```
PP × TP × EP × DP = 8 × 2 × N × M
```

**含义**：
- PP=8：跨 8 个 stage
- TP=2：每个 stage 内 TP=2（单节点 NVLink）
- EP=N：MoE 专家分布到 N 个 rank
- DP=M：M 个副本扩展吞吐

### 4.2 总 rank 数

```
总 rank = PP × TP × EP × DP = 8 × 2 × N × M
```

DeepSeek V4 671B 部署通常用数百到上千张卡。

### 4.3 三维组合的 rank 映射

每个 rank 有 4 个 group：PP group、TP group、EP group、DP group。

```
rank = (pp_idx, tp_idx, ep_idx, dp_idx)

例：PP=2, TP=2, EP=2, DP=1（共 8 个 rank）
rank 0: (0, 0, 0, 0)
rank 1: (0, 0, 1, 0)
rank 2: (0, 1, 0, 0)
rank 3: (0, 1, 1, 0)
rank 4: (1, 0, 0, 0)
rank 5: (1, 0, 1, 0)
rank 6: (1, 1, 0, 0)
rank 7: (1, 1, 1, 0)
```

### 4.4 通信复杂度

每个 step 可能涉及：
- PP send/recv
- TP all-reduce
- EP all-to-all
- DP（推理无通信，训练有）

通信协调复杂，需要 vLLM 的 `parallel_state` 模块统一管理。

---

## 5. 组合并行的设计原则

### 5.1 TP 优先原则

**TP 放在节点内**，因为 TP 通信量大，需要 NVLink 级别带宽。

```
节点 0: TP=2 (rank 0, 1)
节点 1: TP=2 (rank 2, 3)
节点 2: TP=2 (rank 4, 5)
...
```

### 5.2 PP 跨节点

**PP 放在节点间**，因为 PP 通信量小，普通以太网可承受。

### 5.3 EP 配合 MoE

EP 跟 MoE 层绑定，dense 层用 TP。

### 5.4 DP 扩展吞吐

DP 放最外层，每个副本独立。

---

## 6. 通信组管理（vLLM 的实现）

### 6.1 GroupCoordinator 基类

vLLM 用 `GroupCoordinator` 统一管理所有并行组：

```python
class GroupCoordinator:
    def __init__(self, ranks, ...):
        self.ranks = ranks
        self.device_group = new_group(ranks)
        ...
```

### 6.2 多个 Group 实例（不是子类）

vLLM **没有为 PP/TP/EP 创建子类**。所有并行组都是 `GroupCoordinator` 的实例，通过不同的全局变量区分：

```python
# vllm/distributed/parallel_state.py
_PP: GroupCoordinator | None = None    # PP 组实例
_TP: GroupCoordinator | None = None    # TP 组实例
_EP: GroupCoordinator | None = None    # EP 组实例
_DP: GroupCoordinator | None = None    # DP 组实例
```

每个实例包含不同的 ranks，但类型都是 `GroupCoordinator`。

### 6.3 全局获取

```python
get_pp_group()  # 返回 PP 的 GroupCoordinator 实例
get_tp_group()  # 返回 TP 的 GroupCoordinator 实例
get_ep_group()  # 返回 EP 的 GroupCoordinator 实例
```

每个 rank 可以同时属于多个 group。

---

## 7. 组合并行的挑战

### 7.1 调度复杂

scheduler 需要考虑所有并行维度：
- PP：层切分
- TP：每个 PP stage 内部分配
- EP：MoE 层路由

### 7.2 权重加载

不同 rank 加载不同部分的权重：

```
PP Stage 0 加载 layer 0-31 权重
PP Stage 1 加载 layer 32-63 权重
TP rank 0 加载矩阵的左半
TP rank 1 加载矩阵的右半
EP rank 0 加载专家 0-31
EP rank 1 加载专家 32-63
```

vLLM 的 weight loader 处理这种复杂的映射。

### 7.3 通信协调

PP send/recv、TP all-reduce、EP all-to-all 必须正确排序：

```
forward 一层：
1. recv activation from PP prev stage（PP 通信）
2. attention forward（TP all-reduce）
3. MoE forward（EP all-to-all）
4. send activation to PP next stage（PP 通信）
```

### 7.4 容错

某个 rank 挂掉时，怎么处理？
- 训练：通常全停重启
- 推理：复杂，目前 vLLM 没有完善的容错机制

---

## 8. 实际部署案例

### 8.1 DeepSeek V4 671B 推理

```
PP=8 × TP=2 × EP=N × DP=M

总卡数 = 8 × 2 × N × M
典型值：256-1024 张卡
```

### 8.2 vLLM 配置

```bash
# 启动参数
--pipeline-parallel-size 8
--tensor-parallel-size 2
--expert-parallel-size N
# DP 通常通过多实例实现
```

### 8.3 节点规划

```
节点 0-1: PP Stage 0（每个节点 TP=2，EP 内部）
节点 2-3: PP Stage 1
...
节点 14-15: PP Stage 7
```

---

## 9. 小结

- 实际部署几乎都用组合并行（PP + TP + EP + DP）
- PP 切层、TP 切矩阵、EP 切专家、DP 切 batch
- TP 通信量大→放节点内；PP 通信量小→放节点间
- DeepSeek V4 671B 用 PP×TP×EP×DP 数百到上千卡
- vLLM 用 GroupCoordinator 统一管理所有并行组
