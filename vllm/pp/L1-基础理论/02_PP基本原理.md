# 知识点 02：PP 基本原理

> 核心问题：PP 怎么切层？什么是"流水线"？气泡从哪来？

---

## 1. PP 的物理类比

把模型想象成工厂流水线：

```
原料 → 工位1 → 工位2 → 工位3 → 工位4 → 成品
```

- **工位** = stage（PP rank）
- **原料/半成品/成品** = activation（中间激活）
- **传送带** = stage 间的 send/recv 通信

如果只有一个产品（一个 batch），流水线没有优势——总时间 = 各工位时间之和。

流水线的优势来自**多个产品并行流过**：当工位 1 处理产品 2 时，工位 2 正在处理产品 1。

---

## 2. PP 的关键概念

### 2.1 Stage（阶段）

把模型按层切成 N 段，每段叫一个 stage。

```
64 层模型，PP=4：

Stage 0     Stage 1     Stage 2     Stage 3
layer 0-15  layer 16-31 layer 32-47 layer 48-63
```

每个 stage 跑在一个 rank（一个或多个 GPU/NPU）上。

### 2.2 Micro-batch（微批次）

把一个大 batch 切成多个小 batch，让它们流过流水线。

```
batch_size = 32, num_micro_batch = 4
每个 micro-batch 大小 = 8
```

**为什么需要 micro-batch？** 没有它的话，stage 1 必须等 stage 0 处理完整个 batch 才能开始，流水线就退化成串行了。

### 2.3 Forward / Backward

```
Forward：  Stage 0 → Stage 1 → Stage 2 → Stage 3（顺序）
Backward： Stage 3 → Stage 2 → Stage 1 → Stage 0（逆序）
```

训练有 forward + backward，推理只有 forward。

### 2.4 气泡（Bubble）

Stage 空闲等待的时间。

```
Stage 0: |m1|等|等|等|等|等|m1'|
                    ↑↑↑
                  气泡：等 stage 1,2,3 处理完，等梯度传回来
```

气泡是 PP 的核心代价，后续的调度算法都是为了消除/减少气泡。

---

## 3. PP 的执行模型

### 3.1 简化场景：PP=2, micro_batch=1

```
时间:    1    2    3    4
Stage 0: |m1F|...|...|m1B|     F=Forward, B=Backward
Stage 1: |...|m1F|...|m1B|

总时间 = 4
工作时间 = 2（每个 stage 1 forward + 1 backward）
气泡 = 2
```

### 3.2 多 micro-batch：PP=2, micro_batch=2

```
时间:    1    2    3    4    5    6
Stage 0: |m1F|m2F|...|...|m2B|m1B|
Stage 1: |...|m1F|m2F|...|m2B|m1B|

总时间 = 6
每个 stage 工作时间 = 4
气泡 = 2
```

可以看到，micro_batch 增加后，气泡占比下降。

### 3.3 一般公式

对于 PP=N, micro_batch=M：

```
总时间（训练） = 2 × (N + M - 1)
每个 stage 工作时间 = 2 × M
气泡 = 2 × (N - 1)
气泡占比 = (N - 1) / (N + M - 1)
```

**例**：PP=4, micro_batch=4
- 总时间 = 2 × 7 = 14
- 工作 = 8
- 气泡 = 6
- 占比 = 3/7 ≈ 43%

---

## 4. 关键参数及其影响

| 参数 | 增加 | 气泡影响 | 其他影响 |
|------|------|---------|---------|
| PP（stage 数） | ↑ | 气泡占比 ↑ | 单 stage 模型变小，跨节点更多 |
| micro_batch | ↑ | 气泡占比 ↓ | 单 micro_batch 变小，需要足够 batch |
| batch_size | ↑ | 可分更多 micro_batch | 显存压力 ↑ |

**经验法则**：micro_batch ≥ PP × 4，气泡占比降到 20% 以下。

---

## 5. 通信模式

### 5.1 Activation 通信（forward 方向）

```
Stage 0 forward 完成后，把 activation send 给 Stage 1
Stage 1 recv，做自己的 forward，再 send 给 Stage 2
...
```

每次通信是**点对点**（send/recv），不像 TP 那样 all-reduce。

### 5.2 Gradient 通信（backward 方向，仅训练）

```
Stage N-1 backward 完成后，把梯度 send 给 Stage N-2
...
```

也是点对点。

### 5.3 通信量

PP 通信量 = activation 大小 × stage 边界数

跟 TP 比，PP 通信量小得多（TP 每层都 all-reduce，PP 只在 stage 边界）。

---

## 6. PP 的代价

| 代价 | 表现 | 缓解方案 |
|------|------|---------|
| 气泡 | stage 空闲 | 多 micro_batch、async scheduling |
| 内存碎片 | activation 队列管理 | 1F1B 调度（早 backward 早释放） |
| 调度复杂 | 多个 micro_batch 协调 | scheduler 设计 |
| 平衡性要求 | 各 stage 计算量必须均衡 | 层切分要均匀 |

---

## 7. PP 的优势

| 优势 | 原因 |
|------|------|
| 跨节点友好 | send/recv 对带宽要求低 |
| 显存线性扩展 | stage 数 × 单 stage 显存 |
| 实现相对简单 | 比 EP 的 all-to-all 简单 |
| 跟 continuous batching 配合 | 推理场景主流 |

---

## 8. 小结

- PP 的本质：**层切分 + 多 micro-batch 流水**
- 关键参数：PP（stage 数）、micro_batch
- 核心代价：气泡，公式为 `(PP-1) / (PP+micro_batch-1)`
- 通信模式：stage 边界的 send/recv（点对点，低带宽要求）
