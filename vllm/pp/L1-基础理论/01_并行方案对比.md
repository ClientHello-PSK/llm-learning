# 知识点 01：并行方案对比

> 核心问题：单卡放不下大模型时，怎么办？四种并行方案分别切什么？

---

## 1. 出发点：单卡瓶颈

大模型推理/训练的瓶颈有两类：

| 瓶颈类型 | 表现 | 例子 |
|---------|------|------|
| **显存瓶颈** | 模型权重 + KV cache 超出单卡显存 | 671B 模型 ≈ 1.3TB（bf16），单卡 80GB 远远不够 |
| **算力瓶颈** | 单卡算力不足以支撑目标吞吐 | 长序列推理、高 QPS 场景 |

并行就是用多张卡协同解决这两类瓶颈。

---

## 2. 四种并行方案的本质

每种并行都在回答"切什么"，决定了通信模式和适用场景。

### TP（Tensor Parallel，张量并行）

**切什么**：单层内的矩阵

```
矩阵乘法 Y = X × W
TP 把 W 切成多块：W = [W1, W2, ..., Wn]
每张卡算自己那块，然后 all-reduce 合并
```

- **通信**：每层都需要 all-reduce（高频、高带宽）
- **特点**：层内同步，要求低延迟
- **适用**：单节点内多卡（NVLink/NVSwitch 互联）

### PP（Pipeline Parallel，流水线并行）

**切什么**：层（layer）

```
64 层模型，PP=4
Stage 0: layer 0-15
Stage 1: layer 16-31
Stage 2: layer 32-47
Stage 3: layer 48-63
```

- **通信**：stage 边界 send/recv（低频、点对点）
- **特点**：层间异步，可跨节点
- **适用**：跨节点部署、深模型

### EP（Expert Parallel，专家并行）

**切什么**：MoE 的专家

```
256 个专家，EP=8
每张卡放 32 个专家
token 路由到对应专家所在的卡
```

- **通信**：all-to-all（路由分发 + 结果收回）
- **特点**：只对 MoE 模型有效
- **适用**：MoE 模型（DeepSeek V3/V4、Mixtral）

### DP（Data Parallel，数据并行）

**切什么**：batch / 请求

```
8 张卡，每张卡放完整模型副本
每个请求路由到某张卡处理
```

- **通信**：训练需要 all-reduce 梯度；推理无通信
- **特点**：副本独立
- **适用**：模型能单卡放下时扩展吞吐

---

## 3. 四种方案的对比表

| 维度 | TP | PP | EP | DP |
|------|----|----|----|----|
| 切什么 | 矩阵 | 层 | 专家 | batch |
| 通信量 | 高 | 低 | 中 | 训练高/推理无 |
| 通信类型 | all-reduce | send/recv | all-to-all | all-reduce |
| 通信频率 | 每层 | stage 边界 | MoE 层 | 训练每步 |
| 跨节点 | 不推荐 | 推荐 | 推荐 | 推荐 |
| 显存节省 | 部分 | 完整 | 部分 | 不节省（每卡完整副本） |
| 适用模型 | 所有 | 深模型 | MoE | 小模型 |

---

## 4. 选择策略：怎么决定用哪种？

**决策树**：

```
模型能单卡放下？
├─ 是 → DP（多副本扩展吞吐）
└─ 否 → 模型类型？
    ├─ MoE → EP + PP / TP
    └─ Dense → PP / TP
        ├─ 跨节点 → PP 优先
        └─ 单节点 → TP 优先
```

**实际部署中通常是组合**：PP × TP × EP × DP，但每种并行都成倍增加复杂度。

---

## 5. 实际案例：DeepSeek V4 671B

DeepSeek V4 在大规模部署时通常使用：

```
PP × EP × TP × DP
 8   × N  ×  2 ×  M
```

**为什么这么配？**

- **PP=8**：模型太大，必须跨节点切层
- **EP=N**：MoE 架构，专家天然可分
- **TP=2**：单节点内 NVLink 加速
- **DP=M**：剩余卡做副本扩展吞吐

---

## 6. PP 的特殊地位

为什么大型推理服务（vLLM）特别需要 PP？

1. **跨节点友好**：send/recv 对带宽要求低，能在普通以太网上跑
2. **不依赖高速互联**：不需要 NVLink/NVSwitch，普通节点即可
3. **KV cache 自然分布**：每个 stage 维护自己那部分层的 KV cache
4. **跟 continuous batching 配合好**：推理场景的主流调度方式

但 PP 的代价是**气泡**——这是 L1/03 GPipe 调度详解要解决的核心问题。

---

## 7. 小结

- 4 种并行各切各的：TP 切矩阵、PP 切层、EP 切专家、DP 切 batch
- PP 是大模型跨节点部署的主力方案
- PP 的代价是气泡，需要靠调度算法（1F1B、async）来缓解
- 实际部署通常是组合并行，但 PP 几乎是大模型必备
