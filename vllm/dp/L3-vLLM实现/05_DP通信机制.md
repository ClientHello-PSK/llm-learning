# 知识点 05：DP 通信机制

> 核心问题：DP rank 之间需要哪些通信？vLLM 怎么实现？
> 源码位置：`vllm/distributed/parallel_state.py`、`vllm/v1/engine/`

---

## 1. DP 通信的场景

### 1.1 推理场景

| 场景 | 通信 | 频率 |
|------|------|------|
| 非 MoE 模型 | **无** | - |
| MoE 模型（EP 借道） | all-to-all | 每个 MoE 层 |
| 请求路由 | 进程间 RPC | 每请求 |
| 元数据同步 | broadcast_tensor_dict | 每 step |

### 1.2 训练场景（对比）

| 场景 | 通信 | 频率 |
|------|------|------|
| 梯度同步 | all-reduce | 每 step |
| MoE expert 间通信 | all-to-all | 每 MoE 层 |
| 优化器状态 | broadcast | 偶尔 |

---

## 2. 非 MoE 推理：完全无 DP 通信

### 2.1 通信路径

```
请求 → DP rank 0 (或 1, 2, ...)
   ↓
   TP rank 0: forward 一部分
   TP rank 1: forward 另一部分
   TP all-reduce
   ...
   ↓
   输出
   ↓
返回

不跟其他 DP rank 通信
```

### 2.2 性能含义

- N 个 DP rank → N 倍吞吐（在请求充足时）
- 单请求延迟跟 DP rank 数无关
- 加 DP rank 几乎无开销

---

## 3. MoE 推理：EP 借道 DP

### 3.1 通信路径

```
请求 → DP rank 0
   ↓
   pre-attention / norm
   ↓
   attention (TP all-reduce within DP rank)
   ↓
   MoE layer:
      Router 决定每个 token 去哪个 expert
      ↓ all-to-all（跨 DP rank）
      每个 expert 在自己 rank 计算
      ↓ all-to-all（结果返回）
   ↓
   post-attention / norm
   ↓
   返回
```

### 3.2 关键通信：all-to-all

```
每张卡有 token 列表 + 每个 token 的目标 expert
all-to-all 把 token 发到目标 expert 所在 rank
每张卡算自己 expert
all-to-all 把结果发回
```

### 3.3 死锁风险

源码：`vllm/distributed/parallel_state.py:1745-1746`

```
all the ranks in the same DP group should generate simultaneously,
i.e. the `generate` call in the same DP group should be called together,
otherwise it will cause deadlock.
```

如果 DP rank A 已经在 forward 等待 all-to-all，而 DP rank B 还没开始 forward，就会死锁。

### 3.4 vLLM 的解决

- scheduler 同步驱动所有 DP rank
- 每个 step 所有 DP rank 同时 forward
- async scheduling 在 MoE 模型下需要额外注意

---

## 4. 进程间 RPC：请求路由

### 4.1 单实例多 DP rank 模式（MP）

```
vllm CLI 主进程
   ↓ 接收 HTTP 请求
   scheduler 决定路由到 DP rank X
   ↓ IPC（共享内存 / 队列）
   DP rank X 的 EngineCore
   ↓
   Worker (TP)
```

### 4.2 路由策略

vLLM 默认路由策略：

- **round-robin**：循环分配
- **least-loaded**：选择负载最低的 rank（基于 KV cache 占用、batch size）
- **first-available**：选第一个有空位的 rank

### 4.3 元数据同步

每个 step，scheduler 算出 attention metadata，broadcast 给所有 TP rank：

```python
# vllm/v1/worker/gpu_worker.py 简化
def execute_model(model_input):
    if not is_driver:
        # 非 driver TP rank 从 driver rank 接收
        model_input = broadcast_tensor_dict(src=0)
    
    output = model.forward(model_input)
    return output
```

**关键**：broadcast_tensor_dict 是 **TP 组内** 的通信，不是 DP 组。

DP 组之间，每 rank 自己的 scheduler 算自己的 metadata。

## 5. KV Cache 跨 DP rank

### 5.1 默认：不共享

每个 DP rank 有独立的 KV cache pool：

```
rank 0: 请求 A, B, C 的 KV cache
rank 1: 请求 D, E 的 KV cache
```

请求路由到某 rank 后，整个生命周期都在该 rank。

### 5.2 KV cache transfer（实验）

vLLM 有 `kv_transfer` 模块（实验性）：

- P2P KV cache 迁移
- 用于 prefill-decode 分离架构
- 跨 DP rank 共享 KV cache

源码：`vllm/distributed/kv_transfer/`

## 6. metadata 广播

### 6.1 broadcast_tensor_dict

源码：`vllm/distributed/communication_op.py:38`

```python
def broadcast_tensor_dict(tensor_dict, src=0):
    if not torch.distributed.is_initialized():
        return tensor_dict
    return get_tp_group().broadcast_tensor_dict(tensor_dict, src)
```

注意：用 `get_tp_group()`，是 **TP 组内** 的广播。

### 6.2 DP 组内的广播

DP 组内通常通过 scheduler 进程协调：

```
DP rank 0 的 scheduler:
   ↓ 算 metadata
   ↓ 通过 RPC 传给 DP rank 0 的所有 TP rank
   
DP rank 1 的 scheduler:
   ↓ 自己算 metadata
   ↓ 通过 RPC 传给 DP rank 1 的所有 TP rank
```

每个 DP rank 独立算 metadata，不需要 DP 组内的 broadcast。

## 8. 跨 DP rank 的额外通信

### 8.1 全局状态同步

某些场景需要 DP rank 同步状态：

- 模型加载进度
- 服务 ready 信号
- 优雅关闭协调

### 8.2 实现

通过 `dist.barrier()` 或共享内存标志位。

## 9. 通信延迟模型

### 9.1 非 MoE 推理

```
DP 通信延迟 = 0
单请求延迟 = 单副本 forward 时间
```

DP 数量不影响单请求延迟。

### 9.2 MoE 推理

```
DP 通信延迟 = all-to-all 延迟 × MoE 层数
单请求延迟 = 单副本 forward + DP 通信延迟
```

all-to-all 延迟随 EP size 增加而增加：

```
EP=8:  all-to-all ~50μs
EP=32: all-to-all ~150μs
EP=128: all-to-all ~500μs
```

## 10. 通信优化

### 10.1 通信-计算 overlap

async scheduling 让 all-to-all 和其他计算 overlap：

```
传统：
   layer 1 fwd → all-to-all → layer 2 fwd → all-to-all → ...

Async：
   layer 1 fwd ┐
                ├─ all-to-all (与 layer 2 部分 fwd overlap)
   layer 2 fwd ┘
```

### 10.2 弹性 EP（elastic_ep）

源码：`vllm/config/parallel.py:205`

```python
enable_elastic_ep: bool = False
"""Enable elastic expert parallelism with stateless NCCL groups for DP/EP."""
```

允许 EP size 动态变化（动态扩缩容），通过 stateless NCCL group 实现。

### 10.3 DP 数量选择

非 MoE：DP 越多越好（吞吐线性扩展）。

MoE：DP（=EP）增加通信开销，最优 DP 数取决于：

```
DP_optimal = arg max (吞吐 - all-to-all 开销)
```

通常 8-32 是常见最优区间。

## 12. 小结

- 非 MoE 推理：DP 之间**完全无通信**
- MoE 推理：DP（=EP）之间通过 **all-to-all** 通信
- broadcast_tensor_dict 是 **TP 组内** 的广播，不是 DP
- async scheduling 用于通信-计算 overlap
